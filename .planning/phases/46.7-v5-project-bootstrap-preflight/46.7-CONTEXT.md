# Phase 46.7: v5 Project Bootstrap Preflight — Context

## Why This Exists

The bootstrap path should report what project scaffolding exists before init so
the user knows whether memd is seeding from `AGENTS.md`, `CLAUDE.md`, planning
files, or a thinner fallback set. The goal is to keep bootstrap useful even
when the repo is missing some of the usual files.

## Decisions

- Preflight the repo for project scaffolding before seeding the bundle.
- Treat missing agent scaffolding as a weak-context signal, not a blocker.
- Keep memd owning the bootstrap flow directly instead of requiring a separate
  Codex `/init` step.
- Offer a follow-up deeper import path only when the default seed is too thin.

## Discretion Areas

- How detailed the preflight report should be when files are missing.
- Whether the bootstrap surface should mention all candidate files or only the
  ones actually found.
- How much detail should be shown about the fallback import set.

## Deferred Ideas

- Any richer import wizard.
- Deeper project-structure analysis beyond the current file scan.
- Automatic deep import without explicit user request.
