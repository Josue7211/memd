# Phase 46.3: v5 Provider Collision and Canonicalization Controls — Context

## Why This Exists

Once memory is shared across harnesses, the system has to stop one provider
from silently overwriting another provider's truth. Canonical identity needs
to include provider, harness, machine, session, and workspace scope so shared
state stays inspectable instead of collapsing into whichever session wrote
last.

## Decisions

- Keep session identity stable across provider, harness, machine, and
  workspace.
- Preserve contested memory instead of flattening it away.
- Use lease / claim enforcement so active ownership is explicit.
- Surface collision warnings instead of hiding duplicate live identities.
- Keep automatic reconciliation out of the first slice.

## Discretion Areas

- How aggressively collision warnings should be surfaced in status vs
  drilldown views.
- Whether contested memory should be grouped by provider first or by workspace
  first in the human summary.
- How much canonicalization should happen before the user sees the raw conflict.

## Deferred Ideas

- Automatic belief reconciliation.
- Provider-specific trust heuristics beyond the current canonical identity
  model.
- Any UI polish beyond the current warning surfaces.
