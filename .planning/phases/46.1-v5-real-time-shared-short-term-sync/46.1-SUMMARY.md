# Phase 46.1: `v5` Real-Time Shared Short-Term Sync — Summary

## Goal

Make `memd-server` the canonical live state hub for compact shared short-term
memory so Codex, Claude, OpenClaw, and other harnesses can cowork from the
same active state across machines.

## What Changed

1. Shared short-term state now has explicit peer-session records.
   - `memd-schema` carries live peer-session payloads with heartbeat model,
     workspace, agent identity, and session metadata.
   - `memd-server` persists peer sessions in SQLite and exposes list/upsert
     routes for live state publication.
2. The client now publishes and consumes shared live state.
   - heartbeats write best-effort shared session updates to `memd-server`
   - shared awareness prefers the server-backed workspace view
   - remote target-session resolution can find matching sessions through the
     live server view instead of only the local bundle
3. Runtime bootstrap surfaces now point at the shared memory lane.
   - Codex loads memd through the session hook and reload skill
   - Claude’s session hook imports shared memd memory
   - OpenClaw workspace bootstrap points agents at the shared bundle before
     local-only notes
4. The shared state is bounded and structured.
   - the payload stays focused on focus, blockers, next step, claims, ports,
     presence, and handoff/review/help requests
   - the system avoids transcript replay as the sync mechanism

short-term state through `memd-server` instead of relying on isolated local
session memory.
