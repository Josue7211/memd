# Phase 46.1: v5 Real-Time Shared Short-Term Sync — Context

## Why This Exists

The repo already had durable memory, workspace coordination, and runtime
bootstraps, but the active short-term lane was still too local and too easy to
drift apart across machines. This phase makes `memd-server` the canonical live
hub for compact shared state so agents can see the same focus, blockers,
claims, presence, and handoff state without transcript replay.

## Decisions

- Use the server-backed peer-session model as the live short-term source of
  truth.
- Keep the shared state compact and structured instead of syncing raw
  transcripts.
- Treat workspace-scoped state as the active coworking lane and preserve local
  bundle projections for harness-native ergonomics.
- Wire the main agent runtimes to read from the shared lane so Codex, Claude,
  and OpenClaw can pick up the same short-term state.

## Discretion Areas

- Whether runtime wiring should be surfaced in status output, a summary view,
  or both.
- How much of the shared state should be surfaced in the main CLI preview
  versus hidden behind drilldown surfaces.
- Which current-task fields deserve first-class sync versus later expansion.

## Deferred Ideas

- Live push/subscription transport beyond the existing publish/refresh model.
- Any broader cross-project memory promotion logic.
- Automatic conflict resolution beyond explicit collision detection.
