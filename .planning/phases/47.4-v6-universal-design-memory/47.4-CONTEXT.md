# Phase 47.4: v6 Universal Design Memory — Context

## Why This Exists

Design guidance should be reusable memory, not ad hoc prompt text. Memd needs
a typed design lane that can capture `DESIGN.md`-style artifacts and carry
harness-aware metadata so design guidance can move across sessions cleanly.

## Decisions

- Treat `DESIGN.md` and `design/*.md` as first-class bootstrap inputs.
- Keep design guidance inspectable with theme, color, typography, constraints,
  responsive rules, and anti-pattern guidance.
- Classify design artifacts distinctly in the source registry.
- Keep the first slice focused on storage, retrieval, and inspectability.

## Discretion Areas

- How detailed the typed design lane should be before any automatic extraction
  is added.
- Whether design memory should have a separate status view or ride on the
  general memory surfaces.
- How much harness-aware metadata should be visible in the default summaries.

## Deferred Ideas

- Automatic design extraction.
- Richer frontend-specific UI around the design lane.
- Provider-specific design heuristics.
