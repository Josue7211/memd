# Phase 47.2: v6 Token and Context Observability — Context

## Why This Exists

Operators need to see why token budget is being burned before the limit hits.
Memd already surfaces pressure, redundancy, and estimated token footprint in
the hot paths, so this phase formalizes that as a first-class observability
layer.

## Decisions

- Keep observability local, portable, and harness-agnostic.
- Attribute context footprint by source class instead of only by raw size.
- Detect cache cliffs, idle-gap rebuild risk, redundant rereads, and bloated
  shell output patterns.
- Keep the first outputs actionable: compact, fork, prefer compiled artifacts,
  or suppress repeated reads.

## Discretion Areas

- How much source-class detail should be shown inline versus in drilldown.
- Whether warnings should be emitted in status, gap, or both.
- How much provider-specific terminology should be hidden behind generic
  context language.

## Deferred Ideas

- Deep provider-specific telemetry.
- Long-term token trend dashboards.
- Any automatic response beyond warnings and recommendations.
