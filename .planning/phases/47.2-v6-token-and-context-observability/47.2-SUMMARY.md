# Phase 47.2: v6 Token and Context Observability — Summary

## Goal

Make token burn visible before it becomes a failure.

## What Changed

1. Live surfaces already expose pressure signals.
   - `resume`, `refresh`, and `gap` show estimated token use and context
     pressure
   - redundant items and cache-cliff risk are surfaced in the same paths
2. The warnings are actionable.
   - compact instead of continue
   - fork a fresh session with bundle resume
   - prefer compiled artifacts over raw rereads
   - suppress repeated same-session reads when evidence is still fresh
3. The observability model is portable.
   - the same outputs apply across the main harness surfaces without
     provider-specific hacks

## Verification

- existing token-pressure and redundancy outputs are live in the client

## Result

Operators can see why token budget is being burned before limits hit.
