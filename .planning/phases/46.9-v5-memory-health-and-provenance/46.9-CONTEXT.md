# Phase 46.9: v5 Memory Health and Provenance — Context

## Why This Exists

The system should tell operators what memory is missing, stale, duplicated,
inferred, or conflicting, and every imported fact should carry provenance and
confidence so it can be trusted or challenged explicitly.

## Decisions

- Preserve source quality and confidence on memory records.
- Surface stale, contested, derived, and synthetic state rather than hiding it.
- Keep drift and conflict visible in status, maintenance, and ranking views.
- Use provenance as an operator aid, not as silent background metadata only.

## Discretion Areas

- Whether health summaries should rank by severity or by recency.
- How much provenance detail should be shown inline versus in drilldown views.
- Which memory classes should be highlighted first in status surfaces.

## Deferred Ideas

- Automatic confidence calibration.
- Advanced anomaly detection for memory drift.
- Any additional UI polish beyond the existing health surfaces.
