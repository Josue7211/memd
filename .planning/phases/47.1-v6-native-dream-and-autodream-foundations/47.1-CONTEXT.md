# Phase 47.1: v6 Native Dream and Autodream Foundations — Context

## Why This Exists

Dream and autodream should live as native `memd` lifecycle concepts instead of
wrapper-only behavior. The repository already has a native consolidation path,
so this phase is about recognizing that substrate and preserving the boundaries
for future automatic consolidation loops.

## Decisions

- Treat `consolidate` as the native substrate for dream-style lifecycle work.
- Keep consolidation queues, accepted-signal intake, and promotion handoff
  points explicit.
- Preserve compatibility with skill, CLI, MCP, and future app surfaces as thin
  entrypoints.
- Keep the first slice about subsystem boundaries and data flow, not UI polish.

## Discretion Areas

- How much scheduling logic belongs in `memd` itself versus in a wrapper
  surface.
- Whether dream/autodream should appear as separate commands later or stay as a
  lifecycle notion behind consolidation.
- How much of the accepted-signal intake should be automated versus operator
  confirmed.

## Deferred Ideas

- Automatic scheduling policy.
- Any UI polish for dream/autodream.
- Advanced experiment consolidation beyond the native lifecycle boundaries.
