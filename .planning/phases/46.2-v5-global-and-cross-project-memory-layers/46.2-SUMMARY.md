# Phase 46.2: v5 Global and Cross-Project Memory Layers — Summary

## Goal

Make `memd` work as one global memory system with project and cross-project
overlays so users do not have to manage separate repo brains.

## What Changed

1. Global and project bundle routing is now explicit.
   - `memd init --global` resolves to `~/.memd`
   - `memd init --project-root <repo>` resolves to `<repo>/.memd`
   - runtime overlay keeps the project lane distinct from the global lane
2. Cross-project / workspace lanes stay separate from project-local truth.
   - shared awareness and workspace-scoped retrieval preserve initiative
     boundaries
   - the active project still outranks unrelated global memory during resume
3. Bootstrap imports now leave behind a source registry.
   - `state/source-registry.json` records imported source paths, hashes, byte
     counts, and line counts
   - the registry gives future refresh logic a fingerprinted basis for
     read-once imports and delta refreshes

## Verification

- `cargo fmt --all`
- `cargo check -p memd-client`
- `cargo test -p memd-client`

## Result

Memd now behaves like a global-first memory substrate with project overlays
instead of a repo-only bundle tool.
