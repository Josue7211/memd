# Phase 46.2: v5 Global and Cross-Project Memory Layers — Context

## Why This Exists

Memd has to work as a global memory add-on, not only as a project-local
bundle. Agents should load a single canonical global memory root, overlay the
current project when one exists, and still retain a clear boundary for
cross-project initiative state.

## Decisions

- Use `~/.memd` as the global root when the user asks for global memory.
- Resolve repo-local project bundles to `<repo>/.memd` when a project root is
  available.
- Keep project memory separate from global memory, but merge them in the
  runtime view.
- Preserve workspace / initiative lanes as a distinct layer so cross-project
  memory can exist without flattening unrelated repositories.
- Record imported bootstrap sources once with hashes so future refresh logic
  can reason about changed sources instead of rereading everything blindly.

## Discretion Areas

- Whether global or project layers should be surfaced first in human status
  views.
- How much of the source registry should be shown to users versus kept as an
  internal import manifest.
- Whether cross-project lanes should appear in the default summary output or
  only in drilldown views.

## Deferred Ideas

- Automatic delta refresh from the source registry.
- More advanced cross-project promotion rules.
- Any UI polish beyond the current command and summary surfaces.
