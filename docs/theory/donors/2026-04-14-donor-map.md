# Donor Map for memd V2

## Why This Exists

Phase `A3` needs one compact map from mined systems to actual `memd` execution.

This doc is that map.

## Donor Summary

| Donor | What `memd` already does | What donor does better | Decision | Exact target phase |
| --- | --- | --- | --- | --- |
| `mempalace` | raw source retention, wake packet concept, atlas direction | retrieval tuning, conversation mining discipline, practical graph population, benchmark proof posture | `steal now` | `F2`, `E2`, `H2` |
| `Omegon` | local-first runtime, Rust core, state/wake concepts | runtime DB + tracked transport split, single startup assembler, one canonical status surface | `steal now` | `K2`, `J2` |
| `Smriti` | cross-session continuity, hive concepts, session identity | compact claims, freshness checks, divergence signal, exact restore/fork boundaries | `steal now` | `L2`, `J2` |
| `supermemory` | multi-harness intent, dashboard/atlas goals, profile/recall surfaces | thin adapter SDKs, reusable graph product surface, simpler public API packaging | `reference only` | `N2`, `I2`, `E2` |
| `Hermes` | procedural learning direction, self-improvement ambitions | tighter improvement loop posture, operational evolution framing | `reference only` | `M2-evo` |

## Port Order

1. `K2` + `L2`
   - compile one canonical `memd state`
   - add claims
   - add freshness
   - add divergence signal
2. `F2`
   - port conversation mining discipline
   - port dedup discipline
3. `E2`
   - productize atlas graph surface
4. `N2`
   - thin adapter SDKs and shared tool contracts
5. `H2`
   - benchmark proof and behavior-change evidence

## Rejection Rules

- Do not flatten `memd` truth model to match donor products.
- Do not copy hosted-first assumptions into canonical truth flows.
- Do not copy benchmark-shaped heuristics as if they were general doctrine.
- Do not let graph/navigation surfaces become the truth layer.

## Bottom Line

`memd` should keep:

- truth doctrine
- typed memory
- correction lineage
- local-first authority

`memd` should port:

- `Omegon` runtime packaging
- `Smriti` coordination primitives
- `mempalace` mining and retrieval discipline
- `supermemory` adapter and graph packaging patterns
