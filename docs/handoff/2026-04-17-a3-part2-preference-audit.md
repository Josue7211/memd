## Root Cause

**Preferences ARE pulled into wake via `context_compact()` with General intent.**
**BUT:** The `snapshot.context.records` in the Durable Truth section is a **flat, opaque list** of compact memory strings. Preferences lose type information in the compact representation and cannot be distinguished from facts/decisions in the wake output.

The wake markdown renderer (`wakeup.rs:151-198`) iterates `snapshot.context.records` and renders them verbatim without filtering or organizing by `MemoryKind`. Result: preferences exist in memory but are rendered as undifferentiated durable truth, with no visual signal that they are preferences rather than facts.

Additionally, the contract test for preference recall (`cli_contract_runtime.rs:143`) references evidence files (`preference-replay.green/red`) that are **never created** — this is a test placeholder, not live code.

## Decision

- **Root cause:** Pulled-but-not-distinguished — preferences are fetched (via General intent) but rendered indistinguishably in the Durable Truth block because the compact format loses the `MemoryKind::Preference` tag.
- **Fix path for Task 11:**
  1. **Extend `ResumeSnapshot`** to add `preferences: Vec<String>` field (parallel to `context.records` but filtered).
  2. **Update `read_bundle_resume()`** (lines 217–228) to call a second retrieval with `intent=Preference` (not General) to pull only preferences separately.
  3. **Add a new block renderer** in `wakeup.rs` after "## Durable Truth" to show preferences distinctly (e.g., "## Preferences" block).
  4. **Update the contract evidence** in `cli_contract_runtime.rs` to write actual `preference-replay.green` marker files during successful preference recalls (integration test scope).
  5. All changes are **in-process** (client-side retrieve + snapshot + render); no backend changes needed.
