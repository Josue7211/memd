# memd Canonical Theory Synthesis

## Canonical Statement

`memd` is a multiharness second-brain memory substrate for the human.

It should connect:

- models
- harnesses
- agents
- tools
- workflows

through one persistent memory system.

The human owns the brain.
Harnesses route through it.

## Product Goal

The goal is not just memory.

The goal is:

- read once
- remember once
- reuse everywhere
- resume flawlessly
- cut repeated context rebuilding
- maintain or improve quality through sharper memory packets

## What We Learned From Competition

### From mempalace

Keep:

- raw truth retention
- strong retrieval over verbatim data
- small wake-up context
- navigable memory topology

Reject:

- metaphor as full architecture
- lossy compression as default path
- treating semantic retrieval as the whole system

### From Hermes

Keep:

- procedural memory
- always-on loops
- lifecycle hooks
- scheduled automation
- checkpoint and rollback thinking

Reject:

- agent-first ownership
- provider abstraction as memory theory
- skills as the whole memory model

## The Canonical 10-Star Model

### Substrate

These are native memory kinds:

1. `Working context`
2. `Session continuity`
3. `Episodic memory`
4. `Semantic memory`
5. `Procedural memory`
6. `Candidate memory`
7. `Canonical memory`

### Control Functions

These are native control functions:

8. `Correction + provenance`
9. `Wake packet compiler`
10. `Hive coordination`

Latency briefing lives inside hive coordination as the fastest warm-start layer and may later include shared-prefix / KV warm-start for compatible model families.

## Ontology Rule

The theory is only stable if these stay separate:

1. memory kinds
2. control functions
3. surfaces
4. backends/helpers

This means:

- `canonical memory` is not `Obsidian`
- `memory atlas` is not canonical truth
- `semantic backend` is not the model
- `wake packet` is not the substrate
- `latency briefing` is not the truth layer

## Memory Atlas

The memory atlas is the multidimensional navigation layer over canonical memory.

It is not the truth itself.

It supports:

- progressive navigation
- region traversal
- neighborhood expansion
- linked context discovery
- zoom from wake packet to raw evidence

It should support pivots by:

- time
- trust
- provenance
- memory type
- lane
- scope
- harness

Atlas objects should be treated as first-class:

- `region`
- `node`
- `link`
- `atlas path`

The atlas is good when moving deeper feels like zooming, not re-searching from scratch.

Starter atlas lanes:

- `Inspiration lane`
- `Design lane`
- `Architecture lane`
- `Research lane`
- `Workflow lane`
- `Preference lane`

Lanes organize memory by domain across kinds.

## Surfaces

These are surfaces, not the substrate:

- wake packet
- memory atlas
- canonical deep dive
- raw evidence
- Obsidian workspace

Obsidian is a first-class human workspace and readable surface, not the control plane and not canonical truth by itself.

## Live Loop

1. capture raw event/artifact/correction
2. update working context
3. update session continuity
4. write episodic memory
5. repair semantic truth
6. update procedural memory
7. compile wake packet

Retrieval should be typed and staged:

1. wake packet
2. session continuity
3. typed retrieval
4. atlas expansion
5. canonical deep dive
6. raw evidence

Native typed retrieval should include:

- resume retrieval
- episodic retrieval
- semantic retrieval
- procedural retrieval
- correction retrieval
- atlas retrieval
- evidence retrieval

Native retrieval primitives should include:

- exact-span
- entity
- temporal
- procedural
- preference/policy
- semantic expansion

Semantic expansion is helper behavior only.
It must never outrank canonical truth.

## Canonical + Promotion Model

Canonical memory is the durable trusted memory surface the system should prefer before raw evidence.

What can become canonical:

- stable semantic truth
- important episodic anchors
- validated procedures

What should not become canonical easily:

- transient local reasoning
- one-off guesses
- unverified summaries
- stale beliefs

Candidate memory exists to prevent premature canonization.

Promotion should follow signal that is:

- repeated
- validated
- corrected into stability
- useful across sessions
- useful across harnesses

Corrections must update semantic truth quickly.
Old truth should remain visible as superseded, stale, or contested.

Candidate memory should stay one top-level layer with typed internal lanes:

- candidate-semantic
- candidate-procedural
- candidate-episodic

Atlas lanes are separate from candidate lanes.
Candidate lanes are about promotion state.
Atlas lanes are about domain organization.

Canonical episodic memory should be reserved for anchor events that explain later work.

## What Makes memd Different

Not:

- raw-only retrieval
- rag-only retrieval
- summary-only memory
- one-agent local memory

But:

- raw-first truth
- typed memory kinds
- compiled wake packets
- procedural learning
- live correction overwrite
- multiharness shared second brain
- atlas navigation over memory
- canonical promotion discipline
- typed retrieval before giant search

Canonical memory is the durable trusted surface.
Candidate memory exists to prevent premature canonization.

## Hive + Latency

The hive should share:

- canonical truth
- relevant semantic memory
- relevant procedural memory

But each agent should keep its own:

- working context
- local active hypotheses
- local scratch state

Cross-agent and cross-harness handoff should move:

- compact task state
- canonical refs
- procedural refs
- trust/provenance state
- next action

Latency briefing is the first hop.
Wake packet is the richer action-ready packet.

Authority should resolve by stack:

1. human correction
2. fresher verified canonical truth
3. stronger promoted memory
4. stronger candidate provenance
5. local harness guess

Long-term, latency briefing may expand into:

- compact semantic brief
- structured memory refs
- prefix reuse hints
- shared-prefix / shared-KV warm-start data for compatible models

## Win Condition

`memd` wins when:

- fresh sessions resume flawlessly
- humans and agents reread far less
- corrections actually change future behavior
- context gets smaller and sharper
- quality stays equal or improves
- many harnesses feel like one brain
- raw truth remains reachable
- procedures get reused instead of rediscovered

And it must prove that across:

- retrieval benchmarks
- session continuity tests
- correction retention tests
- procedural reuse tests
- cross-harness continuity tests
- repeated-context reduction evidence
- atlas navigation tests
- provenance/trust tests

Default scorecard weighting:

- `20%` session continuity
- `15%` correction retention
- `15%` procedural reuse
- `15%` cross-harness continuity
- `15%` raw retrieval strength
- `10%` token efficiency
- `10%` trust + provenance

Quality preservation is a gate, not a bonus.

## Evaluation Rule

`memd` is not judged by one benchmark.

It must prove:

- raw retrieval strength
- session continuity
- correction retention
- procedural reuse
- cross-harness continuity
- token efficiency
- quality preservation or gain
- trust and provenance

Public benchmarks matter.
But `memd` also needs native product benchmarks for:

- fresh-session resume
- correction overwrite
- cross-harness handoff
- procedural reuse
- atlas navigation

The flagship moat benchmark should be:

- fresh session resume plus cross-harness continuation

## Canonical Architecture Message

The memory system should not ask:

- what can we store?

It should ask:

- what should a fresh session know immediately?
- what should stay true across harnesses?
- what should be learned once and reused forever?
- how do we keep raw truth without drowning in it?
