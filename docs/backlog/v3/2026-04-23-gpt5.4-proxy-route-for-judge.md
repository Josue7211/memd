# gpt-5.4 proxy route for public-bench judge + generator

## Problem

J3 V3-floor verification could not produce canonical primaries for LongMemEval,
LoCoMo, or ConvoMem because the openclaw LiteLLM proxy available to the bench
harness does not route `gpt-5.4` (memd's canonical judge + generator model).

As of 2026-04-23, `GET /v1/models` on the proxy returns only:

- `claude-fallback`
- `gpt-5-codex`
- `haiku-manager`
- `kimi-research`

And `POST /v1/chat/completions` with `model=gpt-5.4` returns
`Invalid model name`.

`haiku-manager` is the only usable generator today, and:

- rejects `temperature=0` when called under `gpt-5-codex` (blocks H3 determinism)
- produces 30-token free-form answers that collapse token-F1 (LoCoMo) and
  exact-match (ConvoMem) metrics regardless of retrieval quality
- works fine for structural constraints like MemBench single-letter MC (4
  completion tokens)

As a result, three of four J3 canonical primaries had to stay `replay-pending`
and only MemBench produced a real mc_accuracy number (0.417, ≥0.70 floor
missed).

## Judge-swap disclosure

Upstream LongMemEval pins `gpt-4o-2024-08-06` as the canonical judge. memd
replays with a **judge-swap to `gpt-5.4` via the openclaw codex-lb route**.
This is documented on each method card in `docs/verification/PUBLIC_LEADERBOARD.md`
so reviewers don't misread our numbers as GPT-4o-judged.

The swap is the right call for memd's infra (flat-rate proxy, no per-token
cost, no out-of-band API key) but must be declared wherever we publish the
numbers.

## H3 shipped

- `call_openai_yes_no_grader_cached` with disk-backed response cache at
  `.memd/benchmarks/grader-cache/<sha>.json`
- `MEMD_BENCH_JUDGE_BUDGET_USD` env cap
- pricing table in `crates/memd-client/src/benchmark/public_benchmark.rs:1304`
  already lists `gpt-5.4`, `gpt-5.3-codex`, `gpt-5.3-codex-spark`,
  `gpt-5.2`, `gpt-oss-120b`, `gpt-oss-20b`, `codex-auto-review` as codex-lb
  flat-rate (0.0 / 0.0)
- 2026-04-23 CLI default flip: both `--generator-model` and `--grader-model`
  defaults are `gpt-5.4` (was `gpt-4o-mini` / `gpt-4o-2024-08-06`). See
  `crates/memd-client/src/benchmark/full_eval.rs:150,155` and
  `crates/memd-client/src/benchmark/runtime.rs:1540,1603,1652`. `gpt-5.4-mini`
  is NOT a canonical memd model — user directive 2026-04-23: provision only
  `gpt-5.4` on the proxy.

None of that exercises without the gpt-5.4 route on the proxy.

## Resolution paths

Preferred:

- Provision `gpt-5.4` on the openclaw LiteLLM proxy (services VM, port 4000,
  `/v1`). Do **not** plan around `gpt-5.4-mini` — user directive 2026-04-23:
  canonical judge + generator is `gpt-5.4`. No client code changes required —
  the harness already honors `OPENAI_BASE_URL` + `OPENAI_API_KEY` and the new
  defaults name this model directly.

Acceptable fallback:

- Use a commercial-OpenAI-direct key for J3 rerun only, capped by
  `MEMD_BENCH_JUDGE_BUDGET_USD=50`. Flip the judge-swap disclosure from
  "gpt-5.4 via codex-lb" to "gpt-4o-2024-08-06 direct" for that run (i.e.
  revert to upstream canonical). Verify the cache-hit rate climbs to >95%
  on second pass so the second verification run is near-free.

Out of scope:

- Sidecar-server gpt-5.4 proxy: overcomplicates the topology when the openclaw
  proxy already terminates LiteLLM auth.

- LongMemEval `qa_accuracy` (ICLR 2025, upstream GPT-4o judge, memd judge-swap
  to gpt-5.4) — current row: `replay-pending`, retrieval-diagnostic
  `session_recall_any@5=0.882` (J3).
- LoCoMo `token_f1_avg` (ACL 2024) — current row: `replay-pending`,
  retrieval-diagnostic `evidence_hit_rate@5=0.360` (J3, 100-item).
- ConvoMem `accuracy` exact-match — current row: `replay-pending`,
  retrieval-diagnostic `recall@k=0.950` (J3, 100-item).

Until unblocked, the leaderboard keeps those three rows in
`replay-pending` and the verified canonical count stays at 0/4 (MemBench at
0.417 is verified-but-below-floor; the floor gate itself is the K3 follow-on).
